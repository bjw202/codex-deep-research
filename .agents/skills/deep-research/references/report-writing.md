# Report Writing Style

## 목적

Deep Research의 중간 산출물은 내부 감사 로그이면서 사용자가 직접 읽는 상세 리포트다. 최종 Synthesis는 감사 로그가 아니라 의사결정 문서다. 정확성과 추적성은 보존하되, 독자가 결론, 근거, 한계, 다음 행동을 빠르게 이해할 수 있게 쓴다.

## 공통 원칙

1. 결론을 먼저 쓴다. 각 문서는 첫 화면에서 "무슨 뜻이고 무엇을 해야 하는지"가 보여야 한다.
2. 표 앞에는 해석 문단을 둔다. 표는 근거 저장소이지 결론을 대신하지 않는다.
3. 프로세스 용어를 남용하지 않는다. `Research Adequacy`, `Expansion Queue`, `vendor-only`, `under-researched` 같은 용어는 필요할 때만 쓰고 쉬운 설명을 붙인다.
4. 독자에게 의미 있는 단위로 묶는다. 출처별 나열보다 "어떤 조건에서 맞는가", "무엇이 위험한가", "어디에 적용할까"로 정리한다.
5. 수치는 측정 단위와 조건을 붙인다. 서로 다른 지표를 단일 생산성 수치처럼 비교하지 않는다.
6. 불확실성은 숨기지 않되 결론을 흐리지 않는다. 한계는 "그래서 어떤 결정을 미뤄야 하는가"로 연결한다.
7. 최종 Synthesis에서는 감사 메타데이터를 앞세우지 않는다. 출처 품질, 게이트, 검색 깊이, Queue 상태는 결론을 보조하는 뒤쪽 섹션이나 부록성 섹션으로 보낸다.
8. 표만으로 설명하지 않는다. 표 앞에는 "왜 중요한가"를, 표 뒤에는 "그래서 무엇을 할 것인가"를 짧게 쓴다. 유즈케이스 표는 핵심 항목 3-5개를 별도 문단으로 풀어 설명한다.

## 문체

- 한국어 기본 문장은 짧게 쓴다. 한 문장에 조건을 2개 이상 겹치지 않는다.
- 추상 명사를 줄이고 동사로 쓴다.
  - 나쁨: "검증 가능한 업무 산출물의 생성-검토-배포 흐름을 재설계"
  - 좋음: "에이전트가 초안을 만들고, 테스트와 리뷰를 거쳐 사람이 승인하는 흐름을 만든다"
- 영어 라벨은 필요한 경우 유지하되 첫 등장에 설명한다.
  - 예: `vendor-only`는 벤더나 고객사가 직접 발표한 자료라 독립 검증이 약하다는 뜻이다.
- "높음/중간/낮음" 같은 확신도는 독자가 오해하지 않게 대상도 함께 쓴다.
  - 예: "출처 존재는 높음, 효과 크기 확신도는 중간 이하"
- 영어 약어와 내부 라벨은 독자가 이미 안다고 가정하지 않는다. 처음에는 쉬운 한국어 이름을 먼저 쓰고 괄호 안에 원어를 둔다.
  - 나쁨: "MCP 기반 T3 workflow는 HITL과 audit ledger가 필요하다."
  - 좋음: "외부 도구 연결을 허용하는 업무는 사람이 승인하고 실행 기록을 남겨야 한다. MCP는 이런 도구 연결 표준 중 하나다."
- AI가 짜깁기한 느낌을 줄이는 문장을 피한다. 특히 명사만 길게 이어 붙인 문장, 영어 라벨이 섞인 표제, "A 및 B 기반 C 고도화" 같은 관공서식 표현을 줄인다.
- 같은 구조의 문장을 반복하지 않는다. 각 주요 섹션은 결론, 이유, 예시, 실행 의미 중 하나를 분명히 맡는다.

## 구조 규칙

모든 긴 산출물은 가능하면 다음 흐름을 따른다.

1. **핵심 요약**: 결론 3-5개, 가장 중요한 조건, 가장 중요한 한계
2. **읽는 법 또는 역할**: 이 문서가 전체 리서치에서 무엇을 담당하는지
3. **주요 발견**: 독자가 의사결정에 쓸 수 있는 발견
4. **근거와 한계**: 표, 출처, 검색 로그, 품질 매트릭스
5. **의사결정 영향 또는 다음 액션**: 적용, 보류, 추가 조사 항목

## 산출물별 초점

| 산출물 | 읽기 목표 | 문체 기준 |
| --- | --- | --- |
| Researcher | 한 관점의 결론과 근거를 이해 | 관점별 결론을 먼저 쓰고, 표는 뒤에서 근거로 사용 |
| Journal | 전체 리서치 흐름과 상충점을 빠르게 파악 | 압축하되 "왜 중요한가"를 각 표 앞에 설명 |
| Critic | 리서치를 어디까지 믿을지 판단 | 판정과 필요한 조치를 먼저 쓰고, 결함은 작업 목록처럼 표현 |
| Verifier | 어떤 주장이 어떻게 정정됐는지 확인 | 원문 대조 결과와 문장 강도 변화를 먼저 설명 |
| External Ingest | 사용자 제공 리서치가 무엇을 보강했는지 확인 | 새로 해결된 gap과 검증 필요 항목을 분리 |
| Synthesis | 결론과 실행 방향을 결정 | 첫 1/3은 결론, 사례, 실행 가이드에 집중 |

## 최종 Synthesis 품질 게이트

최종 리포트를 저장하기 전에 별도 읽기 패스를 수행한다. 이 패스는 사실 검증이 아니라 독자 경험 검증이다.

| Gate | 통과 기준 | 실패하면 |
| --- | --- | --- |
| 독자 질문 | 첫 5문단 안에 "그래서 무엇을 해야 하는가"가 답된다 | 요약을 다시 쓴다 |
| 쉬운 언어 | 내부 라벨, 영어 약어, 방법론 용어가 첫 등장에 설명된다 | 쉬운 한국어로 바꾸거나 뒤로 보낸다 |
| 논리 흐름 | 문제 -> 변화 -> 추천안 -> 실행 조건 -> 근거 순서가 보인다 | 섹션 순서를 바꾼다 |
| 유즈케이스 실감 | 핵심 유즈케이스 3-5개는 업무 전/후가 문단으로 설명된다 | 표 밖에 사례 문단을 추가한다 |
| 표 의존도 | 표가 결론을 대신하지 않는다 | 표 앞뒤 해석을 추가한다 |
| 감사 메타데이터 | 검색 깊이, Gate, Queue, 출처 품질이 앞부분을 지배하지 않는다 | 뒤쪽 부록성 섹션으로 이동한다 |
| 문장 자연스러움 | 소리 내어 읽었을 때 번역투와 짜깁기 느낌이 적다 | 명사구를 동사 문장으로 바꾼다 |

## 표 사용 규칙

- 표가 6행을 넘으면 앞에 요약 bullet을 둔다.
- 표의 `메모` 칸에는 raw evidence만 넣지 말고 결론 영향도 쓴다.
- 출처 목록은 문서 뒤쪽에 둔다. 본문 중간에는 핵심 출처만 언급한다.
- 내부 등급표는 필요하지만, 독자용 결론보다 앞에 오지 않게 한다.

## 좋은 요약의 형태

```markdown
## 핵심 요약

- 결론: ...
- 적용 조건: ...
- 가장 큰 반례: ...
- 지금 해야 할 일: ...
- 아직 모르는 것: ...
```

## 피해야 할 형태

- 출처와 수치만 길게 나열하고 결론을 독자에게 맡긴다.
- "조건부", "한계", "불확실성"을 반복해 실제 권고가 흐려진다.
- 표 안에만 중요한 판단을 숨긴다.
- 최종 리포트 본문을 Journal/Critic의 메타데이터 복사본처럼 쓴다.
- "근거를 보존한다"는 이유로 독자에게 필요 없는 프로세스 용어를 초반에 노출한다.
- 유즈케이스를 한 줄짜리 표 항목으로만 제시하고, 실제 업무가 어떻게 달라지는지 설명하지 않는다.
